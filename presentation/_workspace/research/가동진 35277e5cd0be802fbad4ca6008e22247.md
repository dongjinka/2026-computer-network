## 핵심 설명 방향

ION-DTN은 DTN 개념을 실제로 구현한 대표적인 소프트웨어 사례이다.

우리 프로젝트에서는 ION-DTN을 직접 깊게 수정하거나 실험 플랫폼으로 활용하기보다는, **DTN의 실제 구현 사례이자 설명용 참고 모델**로 사용하는 방향이 적절하다.

즉, ION-DTN 파트의 목적은 다음과 같다.

- DTN이 실제로 구현 가능한 구조임을 보여준다.
- Bundle Protocol, store-and-forward, long-delay communication 개념이 실제 시스템에서 어떻게 나타나는지 설명한다.
- NASA의 우주 통신 사례를 통해 DTN의 필요성과 실용성을 보여준다.

## ION-DTN 테스트

## 1. 배경

<aside>
✅

**DTN(Delay-Tolerant Networking)**

네트워크 link가 자주 끊기거나 지연이 큰 환경(우주통신, 재난, 오지 등)에서도 데이터를 안전하게 보내기 위한 프로토콜. 

핵심 아이디어는 store-and-forward — "보낼 길이 없으면 일단 저장해두고, 길이 열리면 그때 보낸다."

</aside>

NASA/JPL이 만든 오픈소스 구현체 ION-DTN을 사용

## 2. 실험 셋업

노트북 한 대(WSL2 Linux)에 DTN 노드 3개를 띄움

[노드 1: 송신]  →  [노드 2: 릴레이]  →  [노드 3: 수신]
ipn:1.1                         ipn:2.0                       ipn:3.1
:4561                          :4562                         :4563   (UDP 포트)

여기서 "노드 2 → 3 link가 끊긴 상황"을 인위적으로 만들어서 DTN이 어떻게 반응하는지 보기

## 3. 용어

- Bundle: DTN의 데이터 단위. (TCP의 "패킷"에 해당하지만 훨씬 큼, 자기 완결적)
- SDR: 각 노드의 영구 저장소(persistent storage). bundle이 디스크에 박혀 있어서 노드가 꺼졌다 켜져도 살아있음.
- In limbo: bplist로 bundle 상태를 봤을 때 표시되는 마커.
    - "limbo" = "이도저도 아닌 대기 상태"
    - 의미: "이 bundle은 보낼 길이 막혀서 SDR에 보관된 채 기회를 기다리고 있다"
    - 길이 다시 열리면 → 자동으로 forward 큐로 빠져나가 전송됨

## 4. 6단계로 진행

│ Step │                 한 일                                           │                         결과                         
│ 1     │ 노드 3개 기동                                               │ 데몬 정상, UDP 4561~4563 listening                
│ 2    │ 전송 전 큐 확인                                             │ 모든 노드 큐 비어 있음                              
│ 3    │ 노드 2 → 3 link 차단 (b plan ipn:3.0)        │ 출구가 막힘                                          
│ 4    │ 노드 1 → 노드 3으로 718 B 파일 전송        │ bundle이 노드 2 SDR에 **"In limbo"**로 저장됨  
│ 5    │ 노드 3 상태 확인                                           │ 큐 0개, 파일 없음 — 도착 안 함                       
│ 6    │ link 복구 (u plan ipn:3.0)            │ 자동으로 forward → 노드 3에 파일 도착, 내용 100% 일치

### owltsim

 ION-DTN에 같이 들어있는 광속 지연 시뮬레이터

- OWLT = One-Way Light Time — 빛이 한 방향으로 가는 데 걸리는 시간
- sim = simulator

### 하는 일

두 노드 사이에 중간 다리처럼 끼워서, 한쪽이 보낸 UDP 패킷을 일정 시간 붙잡고 있다가 반대쪽으로 흘려보냄.

[노드 A]  →  [owltsim]  →  [노드 B]
↑
지연 추가 (예: 480초)

### 쓰는 이유

- 진짜로 화성에 보낼 수는 없으니까, 노트북 한 대 위에서 우주통신 환경을 흉내내는 것
- 참고 OWLT 값:
    - 지구 ↔ 달: 약 1.3초
    - 지구 ↔ 화성: 약 3~22분 (궤도 위치에 따라)
    - 지구 ↔ 명왕성: 약 4~6시간

- 시나리오 C에서 쓸 이유
시나리오 B는 "link가 끊겼다 붙는" 상황을 봤다면, 
C는 "link는 살아있는데 응답이 수 분 뒤에 오는" 상황.

이때 DTN의 다른 강점들 — bundle TTL, contact plan 기반 스케줄링, LTP 재전송

어떻게 동작하는지 볼 수 있음
